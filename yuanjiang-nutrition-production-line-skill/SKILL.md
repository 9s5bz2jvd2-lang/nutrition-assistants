# 圆酱营养科普生产线

> **一个灵台就是一条生产线：总编定题，分身协作，媒体成品，证据回流。**

本 skill 用于把专业营养学、食品安全、国家标准、医学指南、论文、官方发布或新闻资料，生产成「圆酱说营养」风格的科普内容包。它既能给单个 agent 当工作流，也能给灵台网络当分身协作方案。

核心不是“多叫几个模型写稿”，而是：

```text
资料 → 证据门 → 普通人问题 → 大白话 → 海报/图片 → 视频/音频 → 审稿 → 素材包 → 回流
```

---

## 何时使用

典型触发：

- “把这份标准/论文/新闻做成圆酱说营养内容包”；
- “生成海报、图片、短视频、音频、素材包”；
- “结合灵台分身功能做一个完整工作流”；
- “资料获取、证据提取、营养审稿、海报、视频几个 agent 协作”；
- “有工具就直接产出图片/视频/音频”；
- “我做好后要丢给另一个灵台使用”。

---

## 不要何时使用

- 用户问具体疾病“我该怎么吃/用药/治疗” → 查对应疾病食养 skill，并注意转人工红线。
- 用户要搭建完整 RAG 产品/MVP/上线运维 → 用 `nutrition-ai-productization`。
- 用户要通用食养 Agent 人设、红线、eval 方法论 → 用 `food-agent-methodology`。
- 用户要学术论文写作/投稿/综述 → 用 `anti-hallucination-academic-writing` 或 `academic-research`。
- 只有一次简单命令/格式转换 → 用 bash，不要启动分身工作流。

## 生产线总图

```text
总编主控 Orchestrator
│
├─ 0. 任务卡 / 选题卡
├─ 1. 资料获取分身：找原文、拿 PDF、OCR、source ledger
├─ 2. 证据提取分身：强结论-来源表、适用/不适用边界
├─ 3. 营养安全审稿分身：红线、保守表达、转人工判断
├─ 4. 大白话转译分身：普通人问题、误区、正文
├─ 5. 海报文案分身：标题、副标题、三重点、一句结论
├─ 6. 图片生产分身：海报版式 / 插画 / 最终图片
├─ 7. 视频生产分身：脚本 / 分镜 / 字幕 / 最终视频
├─ 8. 音频生产分身：口播稿 / TTS / BGM / 最终音频
├─ 9. 素材包整理分身：文件夹、命名、来源说明、交付清单
└─ 10. 总编验收与回流：发给人类、沉淀 skill/knowledge/eval/gotcha
```

## 工具选择门

| 任务 | 首选身体 | 说明 |
|---|---|---|
| 文件转换、OCR、压缩、批量命名、叠字幕、合成音视频 | bash / 脚本 | 确定性苦力交给程序 |
| 一次性资料扫描、标题候选、审稿初筛 | daemon | 只要结论，不需要长期记忆 |
| 长期食品标准专员、圆酱风格总编、视频脚本专员 | avatar | 反复岗位，需要积累偏好/样例 |
| 图片/视频/TTS/音乐生成服务 | MCP / CLI / 媒体工具 | 有工具就生成文件；无工具则输出 prompt/脚本 |
| 私有项目事实、联系人、路径、长期偏好 | knowledge | 不写进可移植 skill |
| 可复用流程、模板、红线 | skill | 本文件和 assets |

**原则**：不要为了“显得多 agent”而化身。每个分身必须有独立产物、输入版本和验收标准。

---

## Step 0：总编立任务卡

先立卡，再开工。模板见 `assets/production-templates.md`。

任务卡至少回答：

- 选题是什么？
- 资料来自哪里？是否授权检索？
- 目标产物是拆解卡、正文、海报、图片、视频、音频还是完整素材包？
- 是否涉及医学红线/食品安全争议？
- 哪些环节需要分身？哪些用 bash 即可？
- 给人类的汇报节奏是什么？

任务卡写不清，不要急着分身。

---

## Step 1：资料分型与证据门

先判断资料类型：

| 类型 | 优先回答 |
|---|---|
| 国家标准/法规 | 能不能用？用在哪？限量多少？标签怎么写？ |
| 官方发布/指南 | 官方建议是什么？适用谁？不适用谁？普通人怎么做？ |
| 学术论文 | 发现了什么？对象是谁？能否推广？限制是什么？ |
| 新闻/热点 | 哪些是事实？哪些是解读？哪些待确认？ |
| 用户提问/评论 | 真正担心什么？是否需要补资料？是否触发红线？ |

证据门红线：

- 不编造指南、论文、年份、机构、数据、专家共识；
- 不把网页科普伪装成论文；
- 不把单篇研究写成普遍定论；
- 不把搜索摘要当事实；
- 没有来源的强结论必须删除、追问或改弱。

---

## Step 2：内容拆解卡

每个选题必须先有内容拆解卡：

- 资料类型；
- 资料来源；
- 核心结论（≤30 字）；
- 适用范围；
- 不适用范围；
- 普通人最关心的 3–5 个问题；
- 容易误解的 3 个点；
- 需要保守表达的地方；
- 不能说/暂时不能说的地方。

拆解卡是所有后续分身的唯一事实输入。文案、图片、视频、音频分身不得越过拆解卡新增医学事实。

---

## Step 3：内容生产

默认完整输出顺序：

```text
1. 内容拆解卡
2. 核心结论
3. 大白话总结
4. 普通人最关心的问题
5. 容易误解的地方
6. 海报文案
7. 图片产物或图片 Prompt
8. 短视频脚本
9. 视频产物或视频 Prompt
10. 音频口播稿 / 音频产物
11. 审稿清单
12. 发布素材包清单
```

### 大白话要求

- 先说结论，再解释；
- 小学生也能看懂；
- 专业缩写要翻译，不用 ADI/INS/CNS/JECFA/HR/OR/RR/CI 吓人；
- 剂量类比要保守，不把安全上限说成推荐摄入量；
- 不制造恐慌、不羞辱体型、不营销产品。

### 海报要求

固定包含：标题、副标题、三个重点、一句结论、来源说明、二维码区文案。

标题可吸引，但不能恐吓：

✅ “甜蜜素到底能不能吃？”  
✅ “水果里能不能加甜味剂？先看标准怎么说”  
❌ “这些水果千万别吃！”  
❌ “你每天都在吃毒！”

---

## Step 4：媒体产出门（图片 / 视频 / 音频）

灵台可以直接产出媒体，但要先判断当前灵台具备什么工具。

### 媒体工具探测

按顺序判断：

1. 当前是否有可用图片/视频/TTS/音乐 MCP 或 CLI？
2. 是否有项目约定的媒体生成 skill（如 minimax-cli、media-creation、vision、listen 等）？
3. 是否有本地脚本可排版、加字幕、合成音频视频？
4. 若没有生成工具，交付 prompt、分镜、字幕、口播稿和素材清单，而不是假装已生成。

### 图片生产规则

有图片生成工具时：

- 先生成**无大段中文的版式/插画/背景**；
- 中文标题、正文、二维码区建议后期排版；
- 输出最终图片路径 + prompt + 文字层清单。

无图片工具时：

- 输出图片 prompt + 版式说明 + 后期文字层。

禁止：让图像模型直接生成大段中文、伪造 logo、水印、伪引用。

### 视频生产规则

有视频生成工具时：

- 先产出脚本、分镜、镜头 prompt；
- 字幕单独生成；
- 视频画面不要塞长中文；
- 输出视频路径、字幕文件、封面建议、prompt 记录。

无视频工具时：

- 输出 30–60 秒脚本、分镜表、镜头 prompt、字幕文本。

### 音频生产规则

有 TTS/音乐工具时：

- 生成口播音频；
- 可生成轻背景音乐，但不得盖过人声；
- 输出音频路径、口播稿、音色/语速/情绪参数。

无音频工具时：

- 输出口播稿、TTS 参数建议、BGM 风格建议。

音频口吻：像营养师温柔解释，不像广告播音，不用“震惊/必须/逆天”。

---

## Step 5：分身协作执行

### 何时并行

证据门通过后，以下可并行：

- 大白话转译；
- 海报标题候选；
- 视频脚本候选；
- 图片/视频 prompt 候选；
- 音频口播稿候选。

### 何时不可并行

- 资料未读完不可先写稿；
- 证据边界不清不可先做媒体；
- 医学红线未判断不可进入发布包；
- 用户要求“只改口吻”时，不开事实拓展分身。

### 分身交接格式

所有分身交付必须包含：

```text
【角色】
【输入版本】
【完成内容】
【强结论与来源】
【不确定项】
【红线风险】
【建议主控保留】
【建议删除或改弱】
【产物路径】如有
【下一步】
```

---

## Step 6：总编验收

总编主控合稿，不机械拼接。

验收顺序：

1. 核对每条强结论是否有来源；
2. 删除分身新增的无来源事实；
3. 改弱夸张、恐慌、羞辱、营销表达；
4. 检查医学红线；
5. 检查媒体文件是否真实存在；
6. 整理发布包和来源说明；
7. 给人类交付“能直接使用的文件/路径/下一步”。

---

## 医学与合规红线

以下内容不直接生成个体化建议，必须转人工或提醒咨询专业人士：

- 化验单、处方单、影像报告解读；
- 孕妇、婴幼儿、严重慢病、肿瘤、肾病、肝病等高风险人群个体方案；
- 用药调整、停药、药物与食物相互作用；
- 极端减肥目标、断食、催吐、疑似进食障碍；
- 明确要求诊断、治疗、处方；
- GLP-1、司美格鲁肽、利拉鲁肽、二甲双胍等药物使用建议。

统一安全句：

> 请您寻求正规、专业医师及临床营养师的帮助，AI 意见仅供参考。

## 验收门

交付前逐项打钩：

- [ ] 任务卡已写清；
- [ ] 证据卡已完成；
- [ ] 强结论可追溯；
- [ ] 分身产物标注输入版本；
- [ ] 没有分身新增无来源医学事实；
- [ ] 医学红线已检查；
- [ ] 图片/视频/音频如声称生成，文件真实存在；
- [ ] 无工具时明确交付 prompt/脚本，而非谎称生成；
- [ ] 内部 prompt、私有路径、聊天 ID、凭证、红线阈值未外发；
- [ ] 发布素材包有来源说明与审稿清单；
- [ ] gotcha/template/eval/knowledge 已回流。

---

## Gotchas

1. **多 agent 变成多嘴乱写。**  
   处理：所有分身只基于同一张证据卡工作。

2. **视觉分身为了画面效果新增事实。**  
   处理：视觉只负责画面，不负责医学结论。

3. **“有工具能生成媒体”被误解成“必须立刻生成媒体”。**  
   处理：先看任务目标和成本；需要成品才生成，只需方案就交 prompt/分镜。

4. **图像/视频模型生成乱码中文。**  
   处理：画面不放长中文，中文后期排版，字幕单独输出。

5. **一次性任务误化 avatar。**  
   处理：一次性用 daemon；反复岗位才 avatar。

6. **交付完成但经验没回流。**  
   处理：每次结束记录：哪个模板稳、哪个红线触发、哪个角色值得长期化。

## 回流

每跑完 3 个选题，更新本 skill：

- 最常用的分身组合；
- 最稳的媒体生成参数；
- 最常触发的证据/医学红线；
- 圆酱最喜欢的标题/语气样式；
- 失败样本进入 eval，私有事实进入 knowledge，可复用流程进入 skill。
