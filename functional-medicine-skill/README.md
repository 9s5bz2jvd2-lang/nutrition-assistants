# 功能医学综合 Skill

> **可移植、可供不同 agent 加载与调用**的功能医学 Skill：把知识路由、配方草案与证据记录拆成可审阅的步骤。LingTai 是一种接入方式，不是唯一运行环境；任何能加载 `SKILL.md` 并引用配套 `reference/`、`templates/` 与验证材料的 agent 都可以按自己的 Skill 机制使用它。它不是普通网页或独立应用；也**不是诊断、处方、治疗替代、疗效保证或法规放行工具**。

## 它解决什么问题？

功能医学相关问题往往同时涉及概念解释、饮食/生活方式建议、营养补充剂或草本候选、文献证据、个体安全信息与目标市场法规。把这些混成一句“有研究支持”很容易造成三类错误：

1. 把动物、传统经验或机制研究越级说成人体疗效；
2. 在药物、过敏、妊娠哺乳、病史、原料身份或目标市场未知时假装结论完整；
3. 把“可讨论的草案”误说成临床、法规或产品上线结论。

本 Skill 以渐进式披露把这些环节拆开：普通概念问题不必加载整套资料；进入配方或个体化问题时，才启用完整证据、身份、风险、法规与人审门控。

## 三种使用模式

| 模式 | 适用问题 | 交付重点 |
|---|---|---|
| **A：普通概念查询** | 功能医学概念、机制、指标、生活方式与章节导航 | 按需加载相关知识层，清楚标注证据与适用边界 |
| **B：配方/补充剂设计** | 组合候选、成分比较、剂量讨论、证据与法规审阅 | 全域证据、原料身份、风险与法规门控；人审后才可能最终化 |
| **C：个性化配方便方** | 需要为专业人员准备可审阅的个体化草案 | 固定的“草案 + 提醒”双栏，以及每个候选的四轨证据包 |

详细的 agent 路由见 [`SKILL.md`](./SKILL.md) 与 [`ROUTING.yaml`](./ROUTING.yaml)。

## Mode C：看得见的两栏与四轨

当任务进入 Mode C，输出不应只是一张“推荐清单”。它至少包含：

### 1. 个性化方案草案

列出候选成分/形式、分阶段使用思路、选择依据与可能替代项。它是供专业人员审阅的草案，不把候选直接说成个人处方或保证有效。

### 2. 提醒与人审清单

集中呈现用药与使用时点、相互作用/不良反应红旗、未知信息、监测与复审时点、尚未覆盖的证据或法规问题，以及需要人完成的动作。

### 3. 每个候选的四轨证据包

| 轨道 | 记录内容 | 不可越过的边界 |
|---|---|---|
| **T1：人体临床** | 系统综述、Meta 分析、RCT 或其他人体研究；人群、终点、局限与来源定位 | 没有合适人体证据时，不能声称人体疗效 |
| **T2：动物 / in vivo** | 物种、模型、给药途径、剂量、周期、终点与局限 | 动物剂量或结果不等于人体剂量或结果 |
| **T3：中医 / 物质身份** | 经典原文定位、药典或官方材料、现代临床研究的分层记录 | 传统材料不能替代现代人体疗效证据 |
| **T4：法规** | 原料 × 产品类别 × 声明目标市场的核对记录 | `unknown` / `not_assessed` 不能被写成允许或合规 |

机制、已知不良反应和潜在相互作用属于**提醒/安全侧栏**，不是疗效证据的第四轨。

## 信息不全时会怎样？

本 Skill 不应因为资料不全就把已经提供的内容丢掉，也不能补造缺失事实：

- 未知药物、过敏、妊娠/哺乳、病史、目标市场或产品类别会明确记为 `UNKNOWN`，并进入提醒与人审待办；
- 草本/中药材缺少学名、部位、制备/提取物、标志物或剂量单位时，会标记为 `BLOCKED_IDENTITY`；可保留草案与证据台账，但不进行假定身份的特异性检索；
- 零结果、访问失败或被阻断的检索也应进入台账；
- 这些缺口阻止的是不当检索、最终化、疗效升级或市场声明，而不是把审阅中的草案伪装成“没有问题”。

## 安全与责任边界

- 不诊断，不替代医疗服务，不把输出当作个人处方；
- 不把机制、动物或传统证据升级为人体疗效；
- 不把人审记录自动解释为发布、临床放行、法规放行或产品上线；
- 高风险人群、药物相互作用、红旗症状与信息缺口必须清楚呈现给合格的人类专业人员；
- 任何国家/地区的法规结论必须回到该市场、产品类别、原料身份和可追溯来源逐项判断。

## 在不同 agent 中使用

这是一个可移植的 Skill 目录。将整个 `functional-medicine-skill/` 连同其 `SKILL.md`、`ROUTING.yaml`、`reference/`、`templates/`、`assets/`、`tests/` 与 `validator/` 保持相对路径一起交给目标 agent 的 Skill 安装/加载机制。

```bash
git clone https://github.com/9s5bz2jvd2-lang/nutrition-assistants.git
# 将 nutrition-assistants/functional-medicine-skill/ 复制或安装到目标 agent 的 Skill 目录
```

- **LingTai 示例：** 可放到 `<agent-workdir>/.library/custom/functional-medicine-skill`，再刷新其 Skill catalog；
- **其他 agent：** 使用各自的 Skill/提示词/文件加载机制读取 `SKILL.md`，并按它的按需引用路径加载相关材料；
- **共同要求：** 不同 agent 的权限、知识库、检索工具和人审流程不同。安装本 Skill 不会绕过任何系统权限、医疗安全或法规审查边界。

## 目录导览

- [`SKILL.md`](./SKILL.md)：agent 入口、触发条件和渐进式披露路由；
- [`ROUTING.yaml`](./ROUTING.yaml)：模式、输入、证据轨道、终局门控和文件地图；
- [`reference/`](./reference/)：身份、证据、法规、风险与个性化流程的按需细节；
- [`templates/`](./templates/)：草案、提醒、证据矩阵与台账模板；
- [`assets/eval-cases.md`](./assets/eval-cases.md)：边界与回归案例；
- [`tests/`](./tests/) 与 [`validator/`](./validator/)：随 Skill 一起交付的结构/合同检查材料。

## 一个匿名示例的阅读方式

一个合格的匿名示例应让读者能同时看到：候选方案草案、未知项与红旗、每个成分的 T1–T4 证据及缺层、来源定位、以及哪些事项必须由人完成。它不应被理解为对任何真实个人的建议，也不应被复制为处方或市场宣传。
