# 功能医学综合Skill — 全域证据 + 法规门控版

## 激活条件

当用户的问题或需求涉及以下主题时，读取本skill：
- 功能医学（定义、历史、方法论、六原则）
- 七大失衡（消化吸收菌群、免疫炎症、解毒转化、激素神经递质、氧化还原、循环运输、结构性）
- Go-to-it诊疗路径、功能医学矩阵和时间轴
- 慢性病的功能失衡根源、非药物干预
- 健康管理vs临床医学的对比、功能医学与中医的关系
- 消化系统疾病（肠漏、SIBO、IBS、GERD、幽门螺杆菌、菌群失调）
- 免疫/炎症（慢性炎症、自身免疫病、食物不耐受、过敏）
- 解毒与生物转化（肝脏I/II相、重金属、环境毒素）
- 激素失衡（HPA轴/肾上腺疲劳、胰岛素抵抗、甲状腺、性激素、PCOS、更年期）
- 氧化应激与线粒体功能
- 血脂异常与心血管功能医学
- 骨质疏松、骨肌结构失衡
- 基因多态性（MTHFR、GSTT1、COMT、APOE等）与营养基因组学
- 同型半胱氨酸（HCY）、维生素D、功能营养素详解
- 设计补剂配方 / 营养素方案 / 营养补充方案 / 处方
- 分阶段营养素干预策略（强化期→维持期）
- 配方后循证验证 / 法规合规评估 / 跨管辖区原料标准

## 安全红线（不可逾越）

- **不诊断**：本skill不提供疾病诊断，不替代临床医生的诊断流程
- **不替代治疗**：配方建议仅供参考，不能替代临床药物治疗和医生处方
- **个体化配方便方（Delta新增）**：当用户输入个人情况后，系统可生成**个性化配方便方供专业人员审阅**，但必须：
  - 区分用户自述数据与已验证临床事实
  - 缺失关键信息时主动询问，不假设
  - 证据/安全/法规上下文不完整时，停在澄清、转介或专业审阅结果
  - 不将动物、机制、传统TCM或法规状态信息转化为人体临床疗效/剂量证明
  - 保留所有现有证据、法规、身份、语料扫描和非最终化门控
- **特殊人群安全门**：以下人群的配方建议必须标注"需在医师指导下使用"：
  - 孕妇及哺乳期女性
  - 儿童（<18岁）及青少年
  - 老年人（>65岁）
  - 严重肝肾功能不全患者
  - 正在服处方药者（潜在药物-营养素相互作用）
  - 已知食物/药物过敏者
- **进食障碍/体重羞耻**：涉及体重管理的建议不得使用羞耻化语言
- **证据边界**：所有建议必须标注证据层级（教材推荐/RCT证据/动物实验/专家共识/传统记录）
- **转介义务**：当用户描述的症状超出功能医学营养范畴时，必须建议其就医
- **引用真实**：不得编造文献、DOI、研究结论；证据查询结果如实呈现，查不到就标注"无证据"
- **不编造法规状态**：不得虚构管辖区的法规授权、产品类别批准或原料合规状态
- **不编造来源**：不得声称"已搜索全部互联网"或"已覆盖所有国家"

> 完整安全条款与局限性声明见 `reference/safety_red_lines.md`

## 树状导航执行规则

1. 从用户输入中提取关键词和健康问题
2. 在树中匹配最相关的分支
3. 只展开到回答问题所需的深度——不要过度加载
4. 如果当前深度不够回答，向下一层深入
5. 如果用户问题跨多个分支，分别走各分支后合并
6. 配方设计分支必须走完Step 1-6全部步骤，其中Step 3b文献发现为强制环节，不可跳过
7. RCT查询在配方生成后自动触发，不要等用户要求
8. Step 0文献发现的搜索轮次视领域文献量而定，至少3轮，确保覆盖教材未收录的有效成分
9. **配方后必须执行四轨证据发现**（人体/动物/TCM/机制），不可跳过
10. **配方后必须执行法规门控**（每个成分 × 每个目标市场 × 每个产品类别），不可跳过
11. **植物/TCM成分身份歧义 → `BLOCKED_IDENTITY`**：保留可审草案与阻塞台账，不可假设默认身份继续
12. **未知目标市场 → 仅输出研究/教育候选**，不可声称合规

# 双模式渐进式披露：普通模式 vs 配方全域门控模式

本skill支持两种运行模式：

## 模式A：普通概念查询（选择性加载）

适用于概念解释、机制查询、单一失衡系统深入等问题。按上方路由表按需加载相关文件即可，不要求遍历全部语料。

## 模式B：配方/补剂设计（全域证据 + 人审）— 强制合约

> **任何用户一旦请求配方、补剂、营养素方案、处方、配方案、营养补充方案或分阶段干预策略，均进入模式B；提供个人健康资料时同时进入模式C。**
> **一次调用必须完成可审草案和自动证据包。** 不要求用户知道或手动催促“下一层”。

### 模式B/C v3 的优先执行次序

1. **接收并标准化。** 保留用户已给资料；任何未给资料写为 `UNKNOWN`，不臆造，也不要求重复填写已知资料。
2. **先生成可审草案。** `draft_generated=true` 是每次非崩溃调用的输出不变量。为全部被请求的营养素和植物候选给出成分/形式、剂量或剂量候选、使用阶段、选择依据及替代项；它不是最终处方、诊断或市场批准。
3. **记录分诊与人审项。** IDNHI、DUSF、DMHR、DRG、DFPR 都保留草案，并把未知资料、红旗、相互作用、证据限制和人审任务放入提醒清单。只有真实记录的人类决定才能到 HRF；HRF 不等于自动发布。
4. **对齐植物/中药身份。** 每个植物/中药候选先记录：中文/通用名、拉丁学名、药用部位、制备/提取物、标志成分或标准化、剂量单位。身份完整者进入对应证据检索；身份不完整者为 `BLOCKED_IDENTITY`：草案保留、歧义和补充项显式列出、该植物的特异性检索不执行，并写一条阻塞台账。
5. **自动执行四条并行证据轨。** 对每个证据合格成分自动执行：
   - T1 人群：SR/MA/RCT；
   - T2 动物：物种/模型/途径/剂量/周期/终点/局限，强制“动物结果和剂量不能推出人体疗效或剂量”；
   - T3 中医/物质：经典原文（版本+定位）、药典/官方标准、现代中医临床研究分列；前两者不作为疗效证明；
   - T4 法规：原料 × 产品类别 × 已声明目标市场。
   每个尝试、零结果、访问失败或身份阻塞都必须有一条查询台账：成分、轨道、来源、查询、日期、结果、原因。`not found` 仅表示该范围内未检出，不表示不存在。
6. **单独组装提醒清单。** 机制、已知不良反应、潜在相互作用和用药时点只进入提醒/风险侧栏，并始终带 `mechanism_is_not_efficacy_proof=true`；它们不占据证据轨，也不能升级为人群疗效。
7. **交付固定两栏 + 证据包。** 输出（A）个性化方案草案；（B）提醒清单；另附四轨矩阵、身份/歧义台账、查询台账、法规矩阵、证据与法规缺口和待审问题。
8. **失败关闭只用于最终化、疗效和市场声明。** 语料扫描、身份、法规或检索缺口阻断最终化、市场就绪/全球合规或疗效升格，绝不删除草案或阻止无关证据轨继续运行。

### 模式C：个性化配方便方（专业审阅便方）

| 步骤 | 自动动作 | 输出 |
|---|---|---|
| 0 接收 | 收集症状、诊断、用药、过敏、生活阶段、市场和类别；缺项写 `UNKNOWN` | 最小化接收台账 |
| 1 草案 | 生成覆盖心理/转诊协同、饮食与代谢节律、营养素、植物候选、睡眠压力、肠道/环境、运动社交和监测的多维方案 | 方案草案列 |
| 2 分诊 | 执行风险分诊但不停止草案 | 提醒列 + 人审项 |
| 3 身份 | 对每一植物/中药做身份元组对齐 | 身份/歧义台账 |
| 4 证据 | 自动执行 T1–T4；任何失败都有查询台账行 | 四轨证据包 + 法规矩阵 |
| 5 审计 | 记录最小必要个人资料、来源、日期、残余风险 | 审计/查询/覆盖台账 |
| 6 人审 | 人审决定保留、删除、组合或最终化 | 不自动完成临床、法律或市场结论 |

## V2 稀疏运行时门控

本候选包只定义路由与运行契约，不把本地试跑路径、固定 chunk 数或审查进度写进可部署 Skill。

- 普通问题仍可按本文件和 `ROUTING.yaml` 的人工路由表稀疏加载原始章节。
- 若要启用 v2 自动稀疏检索，必须先对**全部 canonical chunks** 完成 L0–L3 语义审查，并构建可校验的 `expert-index.v2.json`、`registry.v2.json` 与 `source-manifest.json`。
- 运行时必须从实际构建产物读取语料基数、顺序、路径和 SHA-256；不得使用静态文件数或硬编码 chunk 数代替代。

- 当前试跑完成度与审计证据只记录在候选包外的 `REPORT.md` / `MANIFEST.json`；不可复制进可部署路由。

完整集成契约：`reference/v2-sparse-dual-mode-adapter.md`。
